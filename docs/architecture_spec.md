### 状态定义（AgentState）

工作流的共享状态包含：
- `phase`: 当前阶段（INIT → SPEC_WRITING → SPEC_REVIEW → CODING → CODE_REVIEW → TESTING → COMPLETE）
- `messages`: 消息历史（供 Memory 模块使用）
- `requirement`: 用户输入的原始需求
- `product_spec` / `architecture_spec` / `api_spec`: Spec Architect 输出
- `generated_files`: Developer 生成的文件列表
- `review_result`: Reviewer 的审查结果
- `test_result`: QA 的测试结果
- `max_iterations`: 最大重试次数（默认 3）

## 三、数据流设计

```
用户需求（自然语言）
    │
    ▼
Spec Architect ──→ Product Spec + Architecture Spec + API Spec
    │
    ▼
Developer Agent ──→ 代码文件列表 [{path, content, action}]
    │
    ▼
Reviewer Agent ──→ 审查报告 {score, passed, issues, suggestions}
    │
    ├── 未通过 → 反馈回 Developer (循环)
    │
    ▼ 通过
QA Agent ──→ 测试代码 + 执行结果 {total, passed, failed}
    │
    ├── 失败 → 反馈回 Developer (循环)
    │
    ▼ 通过
Finalize ──→ 最终报告
```

## 四、组件详细说明

### 4.1 Spec Architect Agent
- **职责**：需求分析 → 规格文档生成
- **输入**：自然语言需求 + 项目上下文
- **输出**：Product Spec / Architecture Spec / API Spec（JSON）
- **协作**：输出驱动 Developer 的代码生成

### 4.2 Developer Agent
- **职责**：根据规格生成代码
- **输入**：三份 Spec + 审查/测试反馈（如有）
- **输出**：代码文件列表
- **协作**：接收 Reviewer 和 QA 的反馈进行修复

### 4.3 Reviewer Agent
- **职责**：代码质量与安全审查
- **输入**：规格文档 + 生成的代码
- **输出**：审查报告（评分/问题/建议）
- **协作**：审查未通过 → 路由回 Developer

### 4.4 QA Agent
- **职责**：测试生成与执行
- **输入**：规格文档 + 代码
- **输出**：pytest 测试代码 + 执行结果
- **协作**：测试失败 → 路由回 Developer

### 4.5 Orchestrator
- **职责**：状态机编排 + 条件路由
- **实现**：LangGraph StateGraph
- **路由逻辑**：4 个条件网关（spec/developer/reviewer/qa）

## 五、技术选型理由

| 技术 | 理由 |
|------|------|
| **LangGraph** | 提供 StateGraph 原语，原生支持条件路由和循环，适合多步骤 Agent 工作流 |
| **DeepSeek API** | OpenAI 兼容接口，成本低，中文能力强 |
| **ChromaDB** | 轻量级向量数据库，Python 原生，零配置，适合课程项目 |
| **MCP 协议** | 标准化工具接入协议，加分项，支持未来扩展 |
| **LangFuse** | 开源 LLM 可观测平台，支持 Tracing 和评估 |
| **Streamlit** | 最快的 Python Web UI 框架，适合 Demo 展示 |
| **Docker** | 代码执行沙箱隔离，课程要求 |
